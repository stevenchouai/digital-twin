# 学习闭环

这是数字分身和普通 skill 的关键差别。

目标不是让模型“训练权重”，而是让每次任务都留下可复用资产，使数字分身随着使用变得更稳、更像 Steven。

## 何时触发

- 任何复杂任务结束后
- 用户说“这个流程不错”“帮我沉淀一下”“下次别再踩坑”
- 任务里出现了新的偏好、规则、模板、失败模式

## 输出目标

优先写入：

- `wiki/outputs/agent-learnings/`
- `wiki/outputs/research/`
- 当前 repo 的 workflow docs

如果当前不在知识库工作区，至少在回复里明确列出应沉淀的规则。

## Learning Loop 四问

任务结束前快速回答：

1. 这次任务暴露了什么稳定偏好？
2. 哪一步应该被固定成规则或 checklist？
3. 哪个错误以后应该脚本化避免？
4. 哪个结果值得成为模板、范式或例子？

## 产出格式

推荐文件名：

`wiki/outputs/agent-learnings/YYYY-MM-DD-<topic>.md`

推荐内容：

```md
# <主题>

## Trigger
这类任务在什么情况下出现

## What worked
这次有效的方法

## What to avoid
应跳过或避免的做法

## Durable rule
下次默认沿用的规则

## Follow-up
还需要补什么脚本、模板或文档
```

## 判断门槛

只有满足以下任一条件时才写：

- 对未来 2 次以上任务有帮助
- 能减少重复判断
- 能避免已知错误
- 能让数字分身更贴近 Steven 的真实偏好

不要把每次任务都机械写成流水账。

## 和 Hermes 的区别

借鉴 Hermes 的地方：

- 每次任务都尝试沉淀资产
- 记忆不只靠聊天历史
- 优先写规则、模板、skip 条件

不照搬的地方：

- 不依赖独立 runtime 才能成立
- 不追求完整 agent 平台
- 仍然以 wiki 和文件系统为核心
