# Knowledge Base Growth — 知识库成长

这是一个独立的能力模块，专注于知识库的持续成长、整理与回顾。

它和 wiki-management 的区别是：wiki-management 侧重单次 ingest 的执行流程，而 knowledge-growth 侧重的是**跨时间窗口的理解、聚类和回顾**——帮助 Steven 两个月后回看这段时间的状态时，能快速知道"当时我在想什么、方向是什么、心态如何变化"。

## 触发信号

- "帮我 review 一下最近的知识库"
- "最近两个月我的状态怎么样"
- "帮我整理一下最近写的东西，看看有什么关联"
- "知识库太散了，帮我串联一下"
- "帮我做一下聚类"
- "我最近关注的方向有变化吗"

## 前提

当前工作区必须是 Steven 的 Obsidian 知识库（路径包含 StevenChouAI），或可以访问该知识库。

必须存在：
- AGENTS.md
- wiki/_index.md
- journal/ 目录
- raw/thoughts/ 目录

## 五个核心操作

### 操作 1: 状态感知 (State Sync)

**目标**: 了解 Steven 从上次对话到现在之间发生了什么。

**执行步骤**:

1. 读取 journal/ 中最近的月度日记
2. 读取 raw/thoughts/ 中最近新增的文件（按日期判断）
3. 读取 Dashboard/状态仪表盘.md 引用的 journal/state/ 数据（如果存在）
4. 综合以上信息，提取：
   - **工作表现**: 最近在做什么项目、遇到什么挑战、有什么成就
   - **学习方向**: 最近在学什么、在思考什么、在读什么
   - **情绪状态**: 整体情绪基调（注意：Steven 明确说过"情绪你可以了解但不要原样输出，做判断性学习"）
   - **关注方向变化**: 跟之前相比，重心有没有移动

5. 输出一份状态摘要到 wiki/outputs/state-sync/YYYY-MM-DD-state.md

**注意**:
- Steven 的日记包含中英文混合、口语化表达、语音转文字痕迹，这是正常的
- 不要修改或美化日记内容，只提取和理解
- 情绪部分要做判断性学习：了解但不原样输出负面情绪

---

### 操作 2: 增量消化 (Incremental Digest)

**目标**: 处理自上次 ingest 以来新增的所有未处理资料。

这个操作和 wiki-management 的 ingest 流程一致，但增加了一个步骤：检查新资料与已有知识的关系。

**执行步骤**:

1. 扫描 raw/thoughts/ 和 raw/seeds/ 中没有 ingested: frontmatter 的文件
2. 读取 journal/ 中最近的条目
3. 对每篇新资料：
   - 创建 wiki/summaries/ 条目
   - 更新或创建相关 wiki/concepts/ 页面
   - **新增**: 标注这篇资料与已有哪些 concepts 有关联
   - 添加 ingested: frontmatter
4. 更新 wiki/_index.md 和 wiki/_log.md

**与标准 ingest 的区别**:
- 标准 ingest 只关心"这篇资料说了什么"
- 增量消化还要关心"这篇资料和已有知识的交叉点在哪里"

---

### 操作 3: 主题聚类 (Topic Clustering)

**目标**: 发现分散笔记之间的隐藏联系，帮助 Steven 理解自己思考的脉络。

**执行步骤**:

1. 读取所有 wiki/concepts/ 页面的内容和链接关系
2. 读取所有 wiki/summaries/ 页面
3. 读取最近的 raw/thoughts/ 内容
4. 分析主题聚类：

   **聚类维度**:
   - **观点相近**: 两篇文章表达了类似的核心判断
   - **概念关联**: 两篇文章虽然话题不同，但都引用了同一个底层概念
   - **时间关联**: 同一时期写的内容，反映了当时的关注焦点
   - **矛盾关系**: 两篇文章的观点看似矛盾，需要更深层的辨析

5. 补充缺失的双向链接
6. 输出聚类报告到 wiki/outputs/clustering/YYYY-MM-DD-clusters.md

---

### 操作 4: 时间线回顾 (Timeline Review)

**目标**: 按时间窗口综合所有信息源，生成一份可回顾的状态报告。

**默认窗口**: 最近一个月（可通过用户指令调整）

**执行步骤**:

1. 读取目标时间窗口内的所有 journal/ 条目
2. 读取目标时间窗口内新增的 raw/thoughts/ 文件
3. 读取目标时间窗口内新增的 wiki/outputs/ 文件
4. 读取目标时间窗口内新增的 wiki/concepts/ 更新
5. 读取 journal/state/ 中的量化数据（如果存在）
6. 综合生成时间线回顾

**输出到**: wiki/outputs/reviews/YYYY-MM-review.md

**隐私规则**:
- 这个文件只存在于本地知识库，不对外公开
- 但仍然不暴露具体公司名称、同事名字、薪资数字
- 情绪部分做趋势分析，不原样复述负面情绪细节

---

### 操作 5: 知识图谱健康检查 (Graph Health)

**目标**: 确保 Obsidian 知识图谱的连接密度和质量。

**执行步骤**:

1. 遍历 wiki/concepts/ 所有页面，分析链接关系
2. 检查以下问题：
   - **孤立页面**: 没有任何入链的 concept 页
   - **断裂链接**: 链接指向不存在的页面
   - **缺失链接**: 内容相关但没有互相链接的页面
   - **概念冲突**: 两个 concept 页面的观点矛盾
   - **过度中心化**: 某个 concept 被过多引用，可能需要拆分
   - **缺失 concept**: 在 summaries 中被频繁提及但还没有独立 concept 页的主题
3. 输出健康报告到 wiki/outputs/health-checks/health-check-YYYY-MM-DD.md

---

## 操作组合建议

| 场景 | 推荐操作组合 |
|------|-------------|
| "最近加了很多笔记，帮我整理一下" | 增量消化 → 主题聚类 → 图谱健康检查 |
| "两个月没看知识库了，帮我回顾一下" | 状态感知 → 时间线回顾 |
| "知识库感觉很散，帮我串联一下" | 主题聚类 → 图谱健康检查 |
| "我最近状态怎么样" | 状态感知 |
| "全面整理一次" | 增量消化 → 状态感知 → 主题聚类 → 时间线回顾 → 图谱健康检查 |

## 输出目录约定

wiki/outputs/
  state-sync/         # 状态摘要
  clustering/         # 聚类报告
  reviews/            # 时间线回顾
  health-checks/      # 图谱健康检查
  agent-learnings/    # 学习闭环产物
  qa/                 # 问答产物
  research/           # 研究报告

## 禁止事项

- 不修改 raw/ 下的原始内容（除了添加 ingested frontmatter）
- 不删除任何 wiki 页面
- 不在状态摘要或回顾中暴露具体公司名称、同事名字、薪资数字
- 不原样输出 Steven 的负面情绪——了解但做判断性呈现
- 不编造没有来源支撑的概念页内容
- 不为了"好看"而虚构聚类关系——只报告真实存在的关联
